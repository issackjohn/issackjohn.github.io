When Two Profilers Share a Thread

Profilers have a reputation for being straightforward. You turn them on, they collect samples, and the output is supposed to tell you what the program was doing. In practice, the interesting bugs are often not in the sampling itself, but in the handoff between layers.

In Chromium, that handoff became subtle when the JavaScript Self-Profiling API and internal tracing were active at the same time. Both of them could emit CPU profile streams for the same thread. The problem was not that the sampler was wrong; it was that DevTools had no reliable way to tell which stream a sample belonged to.

The bug

When multiple profiling sources are active concurrently, DevTools can end up merging their streams on the same thread. That sounds harmless until you look at the consequences. A profile can become a blur of mixed samples, and JavaScript attribution gets corrupted. What should have been a precise view of a self-profiling run instead became a case of “which source did this sample really come from?”

The subtlety is that this does not look like a crash. It looks like bad interpretation. The profile is still there, but it is no longer trustworthy. In performance work, that is often worse than an outright failure because it quietly poisons the analysis.

The fix across three layers

The fix ended up being a small but important change in the metadata plane. Instead of treating every CPU profile stream as anonymous, we gave each one a source tag. The flow was:

  • V8 added a CpuProfileSource field to profile events so a stream could be labeled at the point it was emitted.
  • Chromium set that source to kSelfProfiling when the JS Self-Profiling API started a V8 CPU profile.
  • DevTools ingested the optional args.data.source field and selected one stream per thread based on source priority, rather than blindly merging them.
{
  "name": "Profile",
  "args": {
    "data": {
      "source": "SelfProfiling"
    }
  }
}

That small piece of metadata turned out to be enough to preserve correct attribution. The important part was not the machinery of sampling; it was the identity of the stream. Once the source was explicit, DevTools could keep the streams separate and build its models from the right one.

Why this mattered

This issue sat at the border between browser internals and developer tools. It was a classic case of a bug that only appears when the system is doing more than one thing at once. The real failure mode was not a missing profile; it was a profile that looked plausible but was logically inconsistent.

The fix made profiling more reliable in exactly the situations where developers are most likely to rely on it: when they are instrumenting a page, tracing internals, or trying to understand JavaScript execution under concurrent activity. In other words: it made the profiler less likely to lie.

Related work

The change landed across three public Chromium projects: DevTools CL 6877206, Chromium CL 6874588, and V8 CL 7122240. The bug is tracked in Chromium issue 375614293.

The lesson is simple: when systems become compositional, identity matters. A profiler is only as trustworthy as the metadata that tells you which stream each sample belongs to.
